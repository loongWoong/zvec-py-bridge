# LLM Semantic Wiki Web 应用 — 实现计划

## 目标
在 `wiki-demo/` 用 Python 实现一个 **Semantic Wiki Runtime**：以 SurrealDB（嵌入式文件模式）为图+文档存储，zvec REST Bridge 负责向量语义检索，Ollama/OpenAI 兼容 LLM 负责编译文档、抽取实体/关系、作为 Agent 回答问题。预置 LLM/向量库主题种子数据，启动即可演示。

## 技术栈决策（已与用户确认）
- **存储**：引入 SurrealDB（`surrealdb==2.0.0` Python SDK，嵌入式 `file://` 模式，零额外 server 部署）
- **向量检索**：SurrealDB + zvec 双库。SurrealDB 存文档/图/元数据/全文检索；zvec 专管向量语义检索（复用现有 REST Bridge + Ollama embedding）
- **LLM 职责**：全功能 — ① 从 raw 源编译生成 WikiDocument（含 summary）；② 抽取 Entity/Tag 并建议关系边；③ 作为 Agent 通过 Tool 回答问题
- **种子数据**：预置 LLM/Transformer/RAG/Embedding/向量库等主题的文档 + 关系 + 实体 + 标签

## 详细实现

### 1. config.py — 配置（复用 demo/kb_data.py 风格）
环境变量驱动：`SURREAL_DB`（默认 `file://wiki_data.db`）、`ZVEC_URL`、`OLLAMA_URL`、`EMBED_MODEL`、`LLM_URL`/`LLM_API`/`LLM_API_KEY`/`LLM_MODEL`、`WIKI_PORT`（默认 8090）。

### 2. db.py — SurrealDB 连接与 Schema
- 用 `surrealdb.Surreal("file://wiki_data.db")` 嵌入式连接，`db.use("wiki","wiki")`。
- 定义 Schema（对应 design.md 第十节）：
  - `DEFINE TABLE document SCHEMALESS`（id/title/summary/content/topic_id/created/updated/status/version）
  - `DEFINE TABLE raw SCHEMALESS`（id/url/author/published/content/collected）
  - `DEFINE TABLE topic SCHEMALESS`（id/name/description）
  - `DEFINE TABLE tag SCHEMALESS`（id/name/parent_id）
  - `DEFINE TABLE entity SCHEMALESS`（id/name/type）
  - `DEFINE TABLE version SCHEMALESS`（版本快照）
  - 关系边表（`TYPE RELATION`）：`belongs_to`(document→topic)、`has_tag`(document→tag)、`mentions`(document→entity)、`references`(document→raw)、`related`/`depends`/`extends`/`implements`/`contradicts`/`supersedes`(document→document)、`child_of`(tag→tag)、`previous_version`(version→version)
  - 全文索引：`DEFINE INDEX ... SEARCH ANALYZER ...` on document.title/content/summary
- 提供全局 `get_db()` 返回已初始化的连接。

### 3. zvec_client.py — zvec 向量检索客户端
精简自 `demo/kb_data.py`，只保留 wiki 需要的能力：
- `register_embedding(dimension)` / `create_collection(dimension)` / `ingest_chunks(chunks)` / `search(query, topk)` / `discover_dimension()` / `check_health()`
- 集合名 `wiki_chunks`，字段 `document_id`/`title`/`heading`/`content`，向量字段 `embedding`（FLAT+IP）。
- chunk 级入库（文档分块后每块一条向量记录），检索返回 chunk 证据片段。

### 4. wiki_runtime.py — Wiki Runtime（核心，对应 design.md 第十一节 Agent Tool）
统一 Tool 接口，LLM 永远调用 Tool 而非直接操作库。分为四个 Service：

**Document Service**：`create_document`/`get_document`/`update_document`/`list_documents`/`delete_document`/`export_markdown`（文档→Markdown 导出，对应 design.md "Markdown 是导出产物"）

**Graph Service**：`relate(src,type,dst,props)`/`neighbors(id,direction,edge_type,depth)`/`related_articles(doc_id)`/`topic_tree()`/`tag_tree()`/`entity_lookup(name)`/`graph_subtree(id,depth)`

**Metadata Service**：`create_topic`/`create_tag`/`create_entity`/`list_topics`/`list_tags`/`list_entities`/`docs_by_topic`/`docs_by_tag`/`docs_by_author`

**Search Service（四路融合，对应 design.md 第九节）**：
- `search_documents(query, topk)` — 融合 ① zvec 向量召回 ② SurrealDB 全文检索 ③ 图遍历（query 命中实体→mentions 反查文档）④ 元数据过滤（解析 author=/topic= 等结构化条件）
- 各路结果按倒数排名融合（RRF）去重，返回统一 `{doc_id, title, score, source[vector|fts|graph|meta], excerpt}` 列表。

**Version Service**：`save_version(doc_id)` 快照当前文档+关系，`list_versions(doc_id)`，`previous_version` 边串联版本链。

### 5. llm.py — LLM 三职责
复用 `demo/agent.py` 的 `_build_llm_request`/`_parse_llm_response`/`_to_provider_messages`/`_clean_answer` 模式（Ollama + OpenAI 兼容双格式）。

**① 编译文档**（`compile_document(raw_content, topic, title_hint)`）：
- 系统提示词要求 LLM 输出 JSON：`{title, summary, content(markdown), entities:[{name,type}], suggested_tags:[...], suggested_relations:[{target_title, type}]}`。
- 调用后由 wiki_runtime 落库为 document + mentions 边 + has_tag 边 + related 边。

**② 抽取实体/关系**（`extract_entities(document_id)`）：
- 读取文档全文，LLM 输出 `{entities:[{name,type}], relations:[{from,to,type}]}`。
- 落库 entity 节点 + mentions 边 + entity 间 related 边。

**③ Agent 问答**（`run_agent(question, max_iterations=6)`）：
- 暴露 wiki_runtime 的 Tool 给 LLM：`search_documents`/`get_document`/`list_documents`/`graph_neighbors`/`related_articles`/`entity_lookup`/`topic_tree`/`tag_tree`。
- 系统提示词要求带引用 `[doc_id] title`，错误守门（tool 返回 error 时如实说明）。
- 返回 `{answer, trace, iterations, elapsed}`，与 demo/agent.py 接口一致。

### 6. seed.py — 种子数据
预置约 10 篇 LLM 主题文档（Transformer/Attention/BERT/RAG/Embedding/向量数据库/HNSW/Agent/Prompt/Fine-tuning），复用 `demo/kb_data.py` 的语料风格。每篇带：
- topic 归属（machine-learning / retrieval / infrastructure）
- tags（LLM/Transformer/Agent 等形成 tag 树）
- entities（Attention/Encoder/Decoder 等）
- 关系边（Transformer extends Attention、RAG depends Embedding、Embedding related 向量数据库 等）
- 同时分块入 zvec 向量库。
提供 `seed_all()` 一键灌入（幂等：先检查是否已存在）。

### 7. app.py — FastAPI Web 应用
**生命周期**：startup 时初始化 SurrealDB schema → 检查 zvec/Ollama 健康 → 若库为空则 `seed_all()` → 注册嵌入函数/建集合（幂等）。

**API 路由**：
- `GET /api/health` — 健康检查（surrealdb/zvec/ollama/llm 状态 + 文档/实体/关系计数）
- `GET /api/documents` / `GET /api/documents/{id}` — 文档列表/详情（含关系邻接）
- `POST /api/documents` — 创建文档（可选触发 LLM 编译）
- `PUT /api/documents/{id}` — 更新（自动 save_version）
- `GET /api/documents/{id}/export` — 导出 Markdown
- `GET /api/graph/{id}` — 图邻接（可指定 direction/edge_type/depth）
- `GET /api/topics` / `GET /api/tags` / `GET /api/entities` — 元数据树
- `GET /api/search?q=...&topk=10` — 四路融合检索
- `POST /api/ask` — Agent 问答（`{question}` → `{answer, trace, ...}`）
- `POST /api/compile` — LLM 编译 raw → wiki 文档
- `POST /api/extract` — 对已有文档抽取实体/关系
- 静态文件挂载 `/` → `static/index.html`

### 8. static/index.html — 单页前端
延续 `demo/static/index.html` 的暖色单色画布 + Geist/Newsreader 字体 + 极简编辑风格。三个主视图（Tab 切换）：
- **文档库**：文档卡片列表，点击展开全文 + 关系邻接图（SVG 力导向小图）
- **知识图谱**：全局图可视化（节点=文档/实体/标签，边=关系），可按类型过滤
- **问答**：输入框 + Agent 回答 + 可折叠的 tool trace
顶部状态栏显示 surrealdb/zvec/ollama 连接状态。搜索框触发四路融合检索，结果标注来源路（向量/全文/图/元数据）。

## 风险与备注
- SurrealDB 嵌入式 `file://` 模式经研究确认可行（Rust/maturin 内置引擎，无需 server 进程）。
- SurrealDB 全文检索/向量算子的精确 SurrealQL 语法因官方文档站点访问超时未能逐字在线核对，实现时将基于 2.x 已知语法编写，若个别算子不符则回退为 SurrealQL `WHERE ... CONTAINS` 子串匹配作为全文检索降级方案，不影响四路融合整体架构。
- LLM 编译/抽取依赖 Ollama/OpenAI 端点可用；若 LLM 不可达，文档库与检索仍可工作（仅编译/抽取/问答功能不可用），前端会明确提示。
