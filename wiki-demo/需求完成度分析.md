# AI 推理引擎 — 需求完成度分析

> 对照文档：`AI推理引擎.md`（设计）
> 实现目录：`wiki-demo/`（Phase 1-3 自报完成，Phase 4 待定）
> 分析日期：2026-07-23

---

## 一、总体结论

**完成度：约 82% / MVP 四阶段中 Phase 1-3 主体落地，Phase 4 未启动。**

实现是一个**真实可运行的 Agentic RAG + 本体引导系统**，技术方向与设计文档高度一致，离线四索引（向量/全文/图/元数据）、在线概念定位、本体展开、多因子 Re-Rank、闭环 Agent 均在代码中**真实存在并相互串联**。DEVELOPMENT_PLAN.md 自报的进度基本属实，不是空壳。

但有 **3 处实质性偏差**会直接影响"本体引导"这一核心卖点的效果，且自报进度中未体现：
1. **Re-Rank 的"概念距离分"实际失效**（权重 0.30 沦为常量）；
2. **5 策略检索计划只生成不执行**（未真正并行调用）；
3. **Step 9 闭环是软闭环**（prompt 自评估 + 硬约束，无结构化"继续/停止"判定）。

另有多处**简化实现**（Step 1 正则切分替代 tree-sitter、概念标注用关键词匹配而非 LLM），这些在 DEVELOPMENT_PLAN 中已被标记为 Phase 4 待补，属已知范围。

## 三、关键偏差（影响核心价值，需优先修复）

### 偏差 1：Re-Rank"概念距离分"实际失效（高优先级）
- **现象**：`search_documents` 返回的候选只含 `doc_id/title/score/sources/excerpt`，**没有 `concept_ids`**；`run_agent` 调用时也不传 `concept_ids`。
- **后果**：`reranker._concept_distance_score` 对所有候选返回中性 0.3，`β=0.30` 权重变成**所有候选一致的常量**，完全不参与排序区分。Re-Rank 实际退化为"语义 + 结构 + 新鲜度"。
- **设计意图**：本体引导的核心就是"概念距离近的优先"。这一因子失效，意味着**本体对排序几乎没有贡献**，与设计"给 LLM 一张地图让它精准导航"的初衷背离。
- **修复成本极低**：在 `search_documents` 合并候选时，从 SurrealDB `document.concept_ids` 取文档级概念回填到候选；并把 `run_agent` 里的 `target_concept_ids=concept_names` 换成**概念 ID** 列表（当前传的是名字，与候选内的 ID 无法匹配）。

### 偏差 2：5 策略检索计划"只生成不执行"（中优先级）
- `ontology_traversal.generate_search_plan` 能生成 5 条策略，但 `run_agent` 仅把 `{bound_documents} 文档 / {bound_files} 文件` 拼进提示文本，Agent 最终只调 `search_documents`。
- 设计 Step 6/7 期望的"向量路 + grep 路 + 文件路 + 图路 + 调用链路 **并行**"并未发生。当前多路并行仅限 `search_documents` 内部的 4 路融合。
- 影响：本体展开产出的"策略 3 文件定位 / 策略 5 调用链"形同装饰，未真正形成多工具并行召回。

### 偏差 3：Step 9 为软闭环，无结构化评估（中优先级）
- 设计要求 LLM 显式输出 `{decision: answer|continue, reason, missing_info, next_search}` 并据 JSON 决定循环。
- 实现改为"Agent 自己决定要不要再调工具 + 系统层硬约束兜底"。能收敛、不易失控，但**缺失"矛盾检测 → 验证性检索"与"缺失环节识别"的结构化信号**，闭环质量依赖模型自觉。
- 影响：对"登录后偶尔 403"这类需要多跳验证的排障问题，召回深度可能不足。

### 偏差 4：概念标注为关键词匹配，非 LLM（低优先级，已知范围）
- `pipeline._annotate_concepts` 用"概念名是否出现在文档前 500 字"做子串匹配，`ontology_builder` 的 LLM 提议能力未被入库流程调用。
- 结果：本体→文档绑定稀疏（主要来自 `concepts.yaml` 手工绑定），大量文档 `concept_ids` 为空——这也放大了偏差 1 的影响。

## 五、建议下一步（按 ROI 排序）

1. **[高] 修复概念距离因子**：让 `search_documents` 回填 `concept_ids`，`run_agent` 传概念 ID。约 20 行改动，即可让本体真正参与排序——这是"本体引导"卖点的临门一脚。
2. **[中] 激活检索计划执行**：把 `generate_search_plan` 的 file/metadata/graph 策略，在 `run_agent` 里转成真实并行工具调用（而不仅是提示），对齐设计 Step 6-7。
3. **[中] 结构化闭环评估**：新增一个轻量评估 LLM 调用（输入精选 chunk+问题，输出 JSON），替代纯依赖 Agent 自觉。
4. **[低] Phase 4 代码场景**：补 `code_analyzer.py`（tree-sitter 抽函数/类/调用链），作为"结构化索引路"接入 `search_documents`，并对代码库排障类问题做端到端验证（设计示例正是 403 排障）。
5. **[低] LLM 概念标注**：入库时调用 `ontology_builder.propose_bindings` 自动标注 `concept_ids`，解决绑定稀疏问题。
