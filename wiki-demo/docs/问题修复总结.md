## 一、关键偏差修复（4 项）

### ① Re-Rank 概念距离因子失效 → 已修复
**根因**：`search_documents` 返回的候选缺 `concept_ids`；`run_agent` 把概念**名**而非**ID**传给 Re-Rank。
**修复**（`wiki_runtime.py`）：
- 新增 `_fetch_concept_ids(doc_ids)`：批量查询 `document.concept_ids`。
- 新增 `_attach_concept_ids(ranked)`：在 RRF 去重后回填候选的 `concept_ids`。
- `search_documents` 的 Re-Rank 路径与降级路径均调用 `_attach_concept_ids`。
**结果**：`reranker._concept_distance_score` 现在能拿到真实概念 ID，概念距离因子真正参与排序。

### ② 5 策略检索计划只生成不执行 → 已修复
**根因**：Step 6 生成 `search_plan` 后仅作为文本注入，未真正执行本体展开与检索。
**修复**（`llm.py`）：
- 新增 `_execute_search_plan(concept_names, search_plan, question, concept_ids)`：
  真正执行 `ontology_traversal.expand_concepts`（N 跳）→ `build_enhanced_query` → `wr.search_documents` 预召回，返回 `(ontology_context, prefetched_docs)`。
- `run_agent` Step 1 调用它，把预召回文档注入 `all_retrieved_docs` 与概念提示。
- **策略5 [调用链]** 现在通过 `code_route=True` 在预召回中激活（见第七节）。

### ③ 软闭环 vs 结构化评估 → 已修复
**根因**：原 `run_agent` 只靠"是否还有工具调用"决定继续/结束，无结构化充分性评估。
**修复**（`llm.py`）：
- 新增 `EVAL_SYSTEM_PROMPT` + `evaluate_retrieval(question, retrieved_docs, concept_info)`：
  返回 `{decision: answer|continue, reason, missing_info, next_search}`。
- 在非简单路径、且已有检索结果、未强制结束时，插入结构化评估：注入 `[系统·评估]` 续搜/作答提示，设置 `force_answer=True`。
- 与既有"硬约束"（最大轮数、引用守门）并存，形成**硬约束 + 结构化评估**的双层闭环。

### ④ 关键词概念标注 vs LLM → 已修复
**根因**：`pipeline._annotate_concepts` 仅用关键词子串匹配，标注稀疏且不语义。
**修复**（`llm.py` + `pipeline.py`）：
- 新增 `ANNOTATE_SYSTEM_PROMPT` + `annotate_document_concepts(title, content, concept_list)`：LLM 从本体概念列表**多选**标注。
- `pipeline._annotate_concepts` 改为**优先 LLM 多选**，失败/不可达时回退关键词子串（保持降级）。
- 入库（`ingest_file`/`ingest_text`）与 `repair_concept_bindings` 均走此路径，使 `concept_ids` 填充更充分，反哺①的概念距离因子。

## 二、已知简化修复（2 项）

### ⑤ Phase 4 代码 AST / 结构化索引（索引 D）→ 已落地
**目标**：实现 AI推理引擎.md Step 1（代码 AST）与 Step 4 索引 D（结构化索引/代码专用）、策略5 [调用链]，**不依赖 tree-sitter**。
**修复**：
- **新增 `code_analyzer.py`**（零外部依赖，纯正则 + 缩进/括号启发式）：
  - 支持 Python / JS / TS（及 Java/Go/Rust/C/... 通用回退）。
  - 提取 `functions`/`classes`/`symbols`（名称、签名、行号）、`calls`（调用链 caller→callee）、`imports`。
  - `search_symbols(query)`：符号名匹配（精确>前缀>子串>拆词）+ snippet grep 回退。
  - `format_symbol_hits()`：生成可喂给 LLM 的文本索引；`grep_in_analysis()`：按正则 grep 行。
- **`wiki_runtime._code_search(query, topk, target_concept_ids)`**：
  以**概念绑定的 `file_path`**（来自 `bind_concept`）为入口，读取真实代码文件 → 提取符号表/调用链 → 匹配符号或 grep 内容 → 返回带 `file_path`+`function_name` 的结构化命中。
- **`search_documents` 新增第五路 `code_route`**：显式开启且已定位概念时，与向量/全文/图/元数据并行召回；并把 `file_path`/`function_name` 透传给 Re-Rank，使 `reranker._structural_score`（结构相关性因子）真正生效。
- **`pipeline` 入库时落库 `code_symbols`**：`create_document` 新增 `code_symbols` 字段，`ingest_file`/`ingest_text` 对代码文件调用 `code_analyzer.analyze_code` 并持久化（即"索引 D"）。

### ⑥ 种子文档概念绑定 → 复核确认（无需改）
复核 `seed.py` 发现：`seed_all_sync` 已通过实体名经 `ontology._concept_rid(ename)` 将 `concept_ids` 绑定到文档。原分析报告中"种子文档未绑定概念"的判断**不准确**——绑定已存在。因此⑥的主要价值落在 **`repair_concept_bindings`**（针对导入/外部入库文档的概念补全）与运行时激活上。

## 三、新增 API（供验证）

| 端点 | 说明 |
|------|------|
| `GET/POST /api/search?code=1` | 四路融合检索，可叠加第五路代码结构化检索 |
| `POST /api/code-search` | 结构化代码检索（按概念绑定的文件路径定位函数/类/调用链） |
| `GET /api/code-search?q=...&concept_names=用户认证` | GET 版代码检索 |
| `GET /api/documents/{doc_id}/code-symbols` | 返回文档落库时提取的代码符号索引（仅代码文档） |
| `POST /api/repair-bindings?limit=200` | 重新标注现有文档概念绑定（LLM 多选 + 关键词兜底） |

`repair-bindings` 亦可在 `app.py` 设 `REPAIR_BINDINGS_ON_STARTUP = True` 后于启动时后台执行（默认关闭，避免每次启动触发 LLM）。
